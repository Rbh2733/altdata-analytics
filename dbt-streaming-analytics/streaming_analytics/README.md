# streaming_analytics (dbt project)

The dbt project for the synthetic streaming-catalog warehouse. The seed is generated by `../scripts/generate_seed.py`; see the repository-level README for the data's provenance and regeneration command.

## How to run

Everything runs from this directory with the committed profile:

```
dbt build --profiles-dir .
```

That one command seeds `seeds/streaming_tracks_raw.csv`, builds all 8 models, and runs all 60 tests against `streaming.duckdb` in this directory (gitignored; delete it any time and rebuild). `dbt seed`, `dbt run`, and `dbt test` work individually with the same `--profiles-dir .` flag. `profiles.yml` uses a relative database path, so run dbt from this directory.

## Architecture and lineage

```
seeds/streaming_tracks_raw.csv          85,000 rows, 1 per track
    |
    v
stg_streaming_tracks          (view)    type casts, date parsing, boolean
    |                                   flag, duration conversions
    +-> int_artist_metrics    (view)    aggregate to artist grain
    +-> int_genre_performance (view)    aggregate to genre grain
    +-> int_label_catalog     (view)    aggregate to label grain
    |
    v
dim_artists                   (table)   from int_artist_metrics; tiers + surrogate key
dim_genres                    (table)   from int_genre_performance; market share,
                                        audio-character classification
fct_track_performance         (table)   from staging + both dims; atomic track grain,
                                        FK surrogate keys, derived tiers and flags
rpt_label_market_share        (table)   from int_label_catalog; pre-aggregated
                                        label report for direct BI consumption
```

Materialization policy (set in `dbt_project.yml`): staging and intermediate are views (always fresh, cheap at this scale), marts are tables (query performance for the dashboard and ad-hoc SQL).

Design choices worth noticing:

- Surrogate keys are MD5 hashes of the natural key (artist name, genre), so fact-to-dimension joins survive cosmetic changes to the natural key columns.
- `fct_track_performance` keeps low-cardinality attributes (genre, label, country) as degenerate dimensions alongside the FKs, trading width for join-free filtering.
- Tier fields (`stream_tier`, `performance_tier`, `career_stage`, `competitive_tier`) are pre-computed in the marts so dashboards filter on indexed values instead of re-deriving buckets.

## Tests

60 tests run in `dbt build`: 50 generic tests declared in the schema yml files (unique, not_null, accepted_values, relationships) and 10 singular SQL tests in `tests/`. What each singular test asserts:

| Test | Assertion |
|---|---|
| `assert_no_negative_streams` | `stream_count` is never negative (streams are cumulative counts) |
| `assert_popularity_in_valid_range` | popularity stays on its 0 to 100 integer scale |
| `assert_release_dates_in_range` | every release year falls in the dataset's 2015 to 2025 window |
| `assert_track_duration_reasonable` | durations are between 30 seconds and 60 minutes |
| `assert_audio_features_in_range` | danceability, energy, instrumentalness in [0, 1]; tempo in [0, 300]; loudness in [-60, 0] |
| `assert_artist_career_timeline_valid` | no artist's first release year is after their latest |
| `assert_genre_market_share_sums_to_100` | genre stream shares sum to 100 within 0.5pp rounding tolerance |
| `assert_label_market_share_sums_to_100` | label stream shares sum to 100 within the same tolerance |
| `assert_fact_has_valid_dimension_keys` | every fact row carries non-null, non-empty artist and genre keys (catches join failures the relationships test can miss, such as empty strings) |
| `assert_stream_totals_consistent_across_layers` | total streams agree exactly between staging and each intermediate aggregate (no rows lost or double-counted between layers) |

The generic and singular layers are deliberately complementary: schema tests catch structural drift (a duplicated key, an unexpected genre value), singular tests catch logic drift (an aggregation that silently drops rows would fail the cross-layer reconciliation even though every schema test still passes).

## Documentation

Model and column descriptions live in `models/*/_*_schema.yml`. Render them:

```
dbt docs generate --profiles-dir .
dbt docs serve --profiles-dir .
```

## Utilities

`validate_data.py` (run from this directory after a build) prints ten analytical spot-checks: row counts per model, market shares, tier distributions, audio fingerprints, and a cross-layer totals check that mirrors the singular test.
